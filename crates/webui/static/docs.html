Architecture Overview

CortexOS is built as a modular, event-driven system with multiple specialized crates:

cortex-core
Event system, runtime, backpressure policies, and core capabilities. The foundation everything builds upon.
cortex-grid
P2P networking layer. Handles LAN discovery, Kademlia DHT, relay mesh, and the new Pipeline Parallelism system.
cortex-node (cortexd)
The main daemon that runs on each device. Handles task execution, skill registration, and network participation.
cortex-webui
Web interface for interacting with the network. Includes chat UI, admin dashboard, and this documentation.

How It Works
1. User sends prompt
2. HEAD node tokenizes & processes first layers
3. Hidden states flow to MIDDLE nodes
4. TAIL node produces final output
5. Result returned to user

Node Discovery

Nodes find each other automatically using two discovery mechanisms:

Local Network Discovery

Uses UDP multicast on port 7077 to find peers on the same network.


Property Value
Protocol UDP Multicast
Port 7077
Multicast Address 224.0.0.251
Broadcast Interval 5 seconds
Peer Timeout 120 seconds
Kademlia Distributed Hash Table

For wide-area peer discovery across the internet using libp2p.


Property Value
Protocol libp2p Kademlia
Transport TCP + Noise encryption
Bootstrap Random walk + stored peers
Bucket Size (k) 20
AirTag-Style Relay Mesh

Anonymous message relay for nodes behind NAT or firewalls.


  • End-to-end encryption (X25519 + ChaCha20-Poly1305)
  • Rotating identities for privacy
  • TTL/hop count limits
  • DHT bulletin board for offline messages

Task Execution Flow

1. Task Submission
User sends prompt via Chat UI or API endpoint POST /api/tasks/pipeline
2. Pipeline Build
PipelineCoordinator queries PeerStore for compute-capable nodes, assigns HEAD/MIDDLE/TAIL roles
3. Task Routing
Task sent to HEAD node via TCP on port node_port + 1000
4. Layer Processing
Each node processes its assigned layers, passes hidden states to next node
5. LLM Inference
TaskServer calls Ollama API (localhost:11434) for actual LLM completion
6. Response Return
Result flows back through pipeline, returned to user with timing metadata
// Task request structure
struct TaskRequest {
    task_id: String,      // Unique identifier
    skill: String,        // "llm", "math", "coding"
    payload: String,      // User's prompt
    from_node: String,    // Origin node ID
}

// Task response structure
struct TaskResponse {
    success: bool,
    task_id: String,
    result: Option<String>,
    execution_time_ms: u64,
}

API Reference

GET /api/status
Returns node status, peer count, uptime
GET /api/peers
List all discovered peers with capabilities
GET /api/stats
Network statistics (peers, tasks, skills)
GET /api/pipeline/status
Current pipeline configuration and nodes
POST /api/tasks/delegate
Send task to single best node
POST /api/tasks/pipeline
Execute task through distributed pipeline
# Example: Run distributed inference
